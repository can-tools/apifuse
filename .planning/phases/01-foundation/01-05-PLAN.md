<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client -> FastAPI app | All HTTP requests cross into the app at this boundary |
| app_env setting -> middleware config | apifuse_settings.app_env (from environment) determines error detail level |
| CORS origin header -> CORSMiddleware | Origin header from browser/client is processed by CORSMiddleware |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05-01 | Information Disclosure | apifuse_http_exception_handler in dev mode | accept | Dev mode returns exc.detail which is the string passed to HTTPException; this is intentional and expected; D-05 defers restrictions to later phase |
| T-01-05-02 | Information Disclosure | apifuse_http_exception_handler in prod mode | mitigate | ErrorHandlingMiddleware (not this handler) controls 500 message; HTTPException handler returns exc.detail which is developer-controlled (not user-supplied); acceptable per D-07 |
| T-01-05-03 | Elevation of Privilege | CORSMiddleware allow_origins=["*"] | accept | D-06 explicitly allows open CORS in Phase 1 (local-first); restriction deferred to security hardening phase; documented in code comment |
| T-01-05-04 | Spoofing | allow_credentials=True with allow_origins=["*"] | mitigate | Note: allow_credentials=True with allow_origins=["*"] is rejected by browsers per CORS spec (browsers require explicit origin for credentialed requests); this combination is safe in practice but should be revisited when CORS is restricted in a later phase; inline code comment on the allow_credentials=True line documents this for future maintainers |
| T-01-05-05 | Tampering | APP_ENV env var -> is_dev flag in ErrorHandlingMiddleware | mitigate | apifuse_settings uses pydantic-settings type coercion; APP_ENV is a string field with default "development"; an attacker controlling the environment could set APP_ENV=development to get verbose errors, but in a production deployment this is controlled by the deployment config, not user input |
</threat_model>
